# 多类型内容创作工具 – 需求规格说明书 V2.0

**版本**：2.0
**日期**：2026-05-27
**作者**：开发团队
**状态**：初稿

---

## 1. 引言

### 1.1 背景与目标

当前工具（写作助手 v0.1.0）为基于 Tauri 2.0 + Vue 3 + TypeScript 的本地化小说编辑管理软件，功能单一，仅面向小说作者。随着自媒体及多类型写作的普及，用户需要一个能同时管理小说、公众号文章、头条文章等多种项目的统一创作环境，并希望借助大语言模型提升写作效率。

V2.0 的目标是将工具升级为**模块化、可扩展的多类型本地创作平台**，在保持数据本地安全的前提下，融入 AI 辅助能力。

### 1.2 适用范围

本文档描述了 V2.0 版本的功能需求、系统架构优化方向和非功能性需求，供开发、测试、设计人员使用。

### 1.3 术语定义

| 术语 | 说明 |
|---|---|
| 项目类型 | 预定义的创作内容类别，如"小说"、"公众号文章"，每种类型有独立的结构和编辑器配置 |
| 模型适配器 | 封装特定大模型厂商 API 的模块，将厂商差异统一为内部标准接口 |
| Context | 调用 AI 时自动附加的上下文信息，包含文档类型、前后文、项目元数据等 |

## 2. 总体架构优化

### 2.1 模块化拆分

项目从单一小说编辑器重构为以下独立模块，通过清晰接口协作：

| 模块 | 职责 |
|---|---|
| 核心框架 | 窗口管理、菜单系统、快捷键注册、全局状态管理、事件总线、插件加载 |
| 项目类型模块 | 定义各类型项目的元信息、目录结构模板、资源管理模式、类型注册 API |
| 编辑器模块 | 提供纯文本（CodeMirror）、富文本（TipTap/Quill）、Markdown 等多种编辑器视图，支持按项目类型动态切换 |
| 大模型集成模块 | 统一的模型适配器框架、流式响应处理、本地 Key 加密存储、调用限流与重试 |
| 数据管理模块 | 本地项目的 CRUD、配置读写、多类型数据适配、项目合法性校验 |
| 设置与配置模块 | 用户偏好管理、快捷键自定义、大模型 Key 管理界面、皮肤/字体设置 |

### 2.2 模块通信机制

- 各模块通过**事件总线**松耦合通信（如：编辑器内容变更 → 通知状态栏更新字数）
- 核心框架通过**依赖注入**向各模块提供全局服务（文件系统访问、配置读写等）
- 项目类型模块向核心框架**注册**自身元信息，核心框架按需加载

### 2.3 扩展性要求

- **新增项目类型**：只需在项目类型模块中实现 `ProjectType` 接口并注册配置，无需改动核心框架
- **新增大模型**：只需实现 `ModelAdapter` 接口并在集成模块中注册，对上层的 AI 面板和编辑器透明
- **新增编辑器视图**：编辑器模块支持注册自定义编辑器组件，项目类型可指定默认编辑器
- **插件系统**（远期）：预留插件加载机制，允许第三方扩展项目类型和模型适配器

## 3. 功能需求

### 3.1 多项目类型支持

#### 3.1.1 项目创建/打开流程

1. 启动软件 → 显示欢迎界面
   - 左侧操作区：新建项目、打开项目、导入项目
   - 右侧列表：最近项目（最多 10 个，显示项目类型图标、名称、路径、最后打开时间）
2. 点击「新建项目」→ 弹出**类型选择界面**
   - 以卡片网格展示各项目类型（图标 + 名称 + 简介）
   - 首批支持类型：小说、公众号文章、头条文章
   - 预留扩展位：知乎回答、小红书笔记、知识库文档、剧本/脚本
3. 选择类型后 → 进入对应项目的初始化流程（输入名称、选择存储路径等） → 自动创建预定义的项目结构
4. 「打开项目」→ 选择包含 `.writinghelper/project.json` 的目录 → 读取项目类型 → 以对应类型加载项目

#### 3.1.2 类型差异化

| 类型 | 项目结构 | 编辑器特性 | 特有功能 |
|---|---|---|---|
| 小说 | 分卷 → 章节，可拖拽排序 | CodeMirror 沉浸式纯文本/Markdown | 大纲视图、角色管理面板、字数和进度统计、章节重排 |
| 公众号文章 | 单篇文章结构 | 富文本编辑器（兼容微信样式） | 封面图上传、摘要编辑、原文链接、原创/转载声明、手机尺寸实时预览 |
| 头条文章 | 单篇文章结构 | 富文本编辑器（适配头条规范） | 多标题（最多3个）、封面管理（单图/三图模式）、广告设置、分类选择 |
| 知乎回答（扩展） | 单篇回答结构 | 富文本编辑器 | 话题标签、引用问题、赞同/评论模拟 |
| 小红书笔记（扩展） | 单篇笔记结构 | 图文混排编辑器 | 标签管理、图片排版、薯队长风格预览 |
| 知识库文档（扩展） | 多级目录 + 文档 | Markdown + 富文本 | 文档间链接、标签系统、全文搜索 |
| 剧本/脚本（扩展） | 场次结构 | 专用编辑器（对白/动作/场景标题） | 角色对白统计、场景管理、场次拖拽 |

#### 3.1.3 模板系统

- 每种项目类型支持用户创建和保存**项目模板**
- 模板内容包含：预置的目录结构、常用设置（字体、主题）、元数据默认值
- 新建项目时可选择"从模板创建"，跳过重复配置

#### 3.1.4 数据存储

- 每个项目以文件夹形式存储，根目录包含 `.writinghelper/project.json`
- `project.json` 结构（V2 扩展）：

```json
{
  "version": "2.0",
  "type": "novel",
  "name": "项目名称",
  "description": "项目简介",
  "coverPath": "",
  "createdAt": "2026-05-27T00:00:00Z",
  "lastOpenedAt": "2026-05-27T10:00:00Z",
  "typeConfig": {
    "wordCountGoal": 100000,
    "genre": "玄幻"
  }
}
```

- 不同类型可在项目文件夹内保存各自的资源文件（图片、封面等）

**开放/本地模型：**

| 类型 | 说明 |
|---|---|
| Ollama | 支持本地运行的开源模型（Qwen、Llama、Phi、Gemma 等），通过 Ollama API 连接 |
| vLLM / LocalAI | 自托管推理服务，兼容 OpenAI API 格式 |

**统一抽象层：**
- 所有模型对外暴露 `chat(messages, options)` 统一接口
- messages 格式对齐 OpenAI Chat Completions 的 `{role, content}` 结构
- 支持普通请求（返回完整响应）与流式响应（SSE，逐 token 回调）
- 适配器负责将内部标准格式转换为各厂商的原生格式

#### 3.2.2 API Key 管理

- 在「设置 → 大模型」页面，每个厂商有独立的：
  - API Key 输入框（输入时遮蔽显示）
  - 自定义 Endpoint URL（可选，用于代理或私有部署）
  - 默认模型选择下拉框
  - 「测试连接」按钮，点击后发送一条简短请求验证 Key 是否可用
- Key 经 AES-256-GCM 加密后存入本地配置文件（`{config_dir}/writeHelper/ai_keys.json`），**不上传任何服务器**
- 加密密钥由设备指纹派生或用户设置的主密码派生
- 日志中绝不记录 Key 明文，仅在内存中解密后用于 HTTP 请求头
- 支持从环境变量导入 Key（高级用户）

#### 3.2.3 AI 辅助写作功能

在编辑器中选中文本或定位光标后，可通过右键菜单（AI 写作）、快捷键或侧边栏 AI 面板调用以下功能：

| 功能 | 描述 | 输入 | 输出 |
|---|---|---|---|
| 续写 | 基于上下文继续生成内容，保持风格一致 | 自动取光标前 2000 字作为上文 | 流式输出续写内容，可一键插入或复制 |
| 改写/润色 | 调整语气或风格（正式、幽默、简洁、学术、口语化等） | 用户选中的文本 + 选择的风格 | 流式输出改写结果，支持并排对比查看 |
| 扩写 | 按字数或比例扩展选中内容 | 选中文本 + 目标字数（如"扩展到 500 字"） | 流式输出扩写结果 |
| 缩写 | 精简选中内容，保留核心信息 | 选中文本 + 目标字数或比例 | 流式输出精简结果 |
| 生成大纲 | 根据文章主题或已有内容生成章节/段落大纲 | 文章标题 + 已有内容摘要 | 结构化大纲（可插入到大纲面板） |
| 多标题生成 | 为文章生成若干备选标题 | 全文或开头部分 | 5-10 个备选标题列表，可点击选用 |
| 错别字/语法修正 | AI 驱动的错别字和语法检测（替代/补充规则引擎） | 选中文本 | 逐条修改建议（定位 → 应用 → 忽略） |
| 文风仿写 | 以指定作家风格重写内容 | 选中文本 + 风格描述（如"像鲁迅的风格"、"像村上春树"） | 流式输出仿写结果 |
| 角色对话生成 | 根据角色设定生成对话片段 | 角色信息 + 场景描述 | 对话内容，可插入正文 |
| 自定义指令 | 用户输入自然语言指令自由操作文本 | 选中文本 + 自然语言指令 | AI 处理后的结果 |
| 翻译 | 将选中文本翻译为指定语言 | 选中文本 + 目标语言 | 翻译后的文本 |
| 总结 | 概括选中内容或整章要点 | 选中文本或当前章节全文 | 简洁的摘要（可插入到章节开头） |
| 拆解/扩章 | 将一个章节拆分为多个，或将大纲扩展为正文 | 章节内容或大纲条目 | 生成的内容，自动创建为新章节 |

**调用机制：**
- 自动附加 Context：当前文档类型、所在项目元数据、光标前后各 2000 字的上下文、大纲面板中相关条目、角色设定信息
- 支持**流式输出**，实时打字效果，可随时中断
- AI 面板以**侧边栏**形式呈现（可固定或浮动），显示当前对话历史、快捷指令按钮
- 编辑器内选中文本后，浮动弹出 AI 快捷工具栏（类似 Notion AI）

#### 3.2.4 会话管理

- 每次 AI 对话以"会话"为单位，存储在项目目录下 `.writinghelper/ai_sessions/` 中
- 会话记录包含：时间戳、使用的模型、完整的 prompt 和 response、用户反馈（赞/踩）
- 支持查看历史会话、删除会话、从历史会话恢复上下文继续对话
- 会话内容纯本地存储，不上传

#### 3.2.5 注意事项

- 每次调用前在界面上明确提示"正在请求付费 AI 模型 [模型名称]"
- 用户可设置**每日调用上限**和**单次调用 token 上限**，超出后自动拦截并提示
- 网络超时（默认 30 秒）和 API 错误时优雅降级，显示具体错误信息
- 对于免费模型或自建代理，允许填写自定义 Endpoint、自定义请求头、自定义模型名称

### 3.3 编辑器模块升级

#### 3.3.1 多编辑器支持

- **CodeMirror 6**（已有）：纯文本/Markdown，适合小说创作
- **富文本编辑器**（新增）：基于 TipTap，适合公众号/头条等需要排版预览的场景
- **Markdown 预览编辑器**（新增）：左侧编辑 Markdown，右侧实时预览

#### 3.3.2 编辑器统一接口

各编辑器实现统一的 `EditorInterface`：

```typescript
interface EditorInterface {
  getContent(): string
  setContent(content: string): void
  getSelectedText(): string
  insertText(text: string, position?: number): void
  replaceSelectedText(text: string): void
  onContentChange(callback: (content: string) => void): void
  focus(): void
  getCursorPosition(): number
  setCursorPosition(pos: number): void
  // 字数统计
  getWordCount(): WordCount
}
```

#### 3.3.3 编辑器切换

- 项目类型在配置中指定默认编辑器，用户可在设置中覆盖
- 同一项目类型可配置多个编辑器视图（如小说除 Markdown 编辑外，也可切换富文本预览模式）
- 切换编辑器时保持内容同步

## 4. 非功能性需求

### 4.1 安全性

- API Key 仅存本地，使用 AES-256-GCM 加密，密钥派生自设备指纹
- 所有大模型 API 调用使用 HTTPS 传输
- 日志中绝不记录 Key 明文或完整 prompt 内容
- 敏感操作（删除项目文件夹、清空 AI 会话等）需二次确认
- 不收集任何用户数据，不上传任何文件，无需联网即可使用核心写作功能

### 4.2 性能

- 大模型调用**异步执行**，不阻塞主界面 UI 线程
- 流式响应首字显示延迟 ≤ 200ms（取决于网络和模型）
- 编辑器支持 **10 万字以上**单文档流畅编辑
- 冷启动时间 < 3 秒
- 安装包体积 < 30MB
- 空闲内存占用 < 150MB

### 4.3 兼容性

- 支持 **Windows 10/11**、**macOS 12+**、**Linux (Ubuntu 20.04+/Debian 11+)**
- 项目文件与编辑器内容统一 **UTF-8** 编码
- 项目格式向下兼容（V1.0 小说项目可在 V2.0 打开，自动补全 `type` 字段）

### 4.4 可维护性

- 模块间低耦合，每个模块有独立的目录结构、类型定义和测试文件
- 关键接口使用 TypeScript 接口定义，作为模块间的契约
- 各模块有独立的 `README.md` 说明其职责、接口和使用方式
- 遵循 Vue 3 Composition API 规范，组件保持单一职责

### 4.5 本地化

- 界面与文档默认支持中文
- 代码中所有面向用户的字符串集中管理（使用 i18n 方案），为未来国际化预留
- 日期时间格式、数字格式可配置

### 4.6 可访问性

- 支持键盘导航（Tab 切换焦点、Enter 确认、Esc 关闭面板）
- 所有功能可通过键盘快捷键操作（无需鼠标）
- 色彩对比度满足 WCAG AA 标准

## 6. 数据迁移方案

### 6.1 V1.0 → V2.0 项目迁移

1. 打开 V1.0 小说项目时，自动检测 `project.json` 中缺少 `type` 字段
2. 提示用户"此项目为旧版格式，将自动升级为 V2.0 格式"
3. 自动补充：`"type": "novel"`、`"version": "2.0"`
4. 项目结构无需变更（小说类型沿用原有目录结构）

### 6.2 配置文件迁移

- 原有配置项保持不变，新增配置项使用新 key，不影响旧设置
- AI Key 配置为全新文件，不影响现有 `config.json`
